## 1강 데이터베이스
* 데이터베이스 : 데이터들을 모아 통합적으로 관리하는 기술
ex) 로그인의 구현
* SQL[Structured Query Language] : 관계형 데이터베이스 관리 시스템(RDBMS)에서 데이터를 정의하고 조작하기 위한 표준 프로그래밍 언어

* 요구사항 접수 -> 개념적 데이터 모델링 -> 논리적 데이터 모델링 -> 물리적 데이터 모델링 ->데이터베이스에 저장할 수 있게 세팅
* 엔터티[ENTITY] : 업무에 필요한 정보를 저장/관리하기 위한 집합적인 명사 개념적
* 인스턴스 [INSTANCE] : 엔터티 집합 내에 존재하는 개별적인 대상

### 엔터티
 ** 엔터티의 특징 **
  - 반드시 업무에서 필요한 대상이고 업무에 사용될 것.
  - 유일한 식별자로 식별이 가능할 것
  - 인스턴스가 2개 이상일 것
  - 속성이 반드시 2개 이상 존재할 것
  - 관계가 하나 이상 존재할 것

 * 엔터티의 분류(유무형에 따라 분류)
  - 유형[Tangible](물리적 형태가 있는 엔터티) ex_직원 주류 강사
  - 개념[Conceptual](물리적 형태가 없는 엔터티) ex_부서 과먹 계급
  - 사건[Event](업무 수행 중 발생하는 엔터티) ex_강의 매출 주문 상담

 ** 엔터티의 분류(발생시점에 따라 분류) **
  - 기본/키
  - 중심
  - 행위

 ** 명명규칙 **
  - 현업용어 사용 권장 (사람->고객)
  - 약어 X (일매목 ->일별매출정보)
  - 단수명사(직원들->직원)
  - 유일한 엔터티이름
  - 생성의미대로 이름을 부여(연락처목록 -> 직원/고객 연락처 목록)

### 속성[ATTRIBUTE]
업무상 관리하기 위해 의미적으로 더는 부닐되지 않는 최소의 데이터 단위

* 식별자 : 엔터티 내 유일한 인스턴스를 식별할 수 있는 속성의 집합
ex_ 직원ID 겹치지 않으면 OK/이름은 동명이인이 있으면 X

** 속성의 분류(특성에 따른 분류) **
 - 기본 : 업무로부터 추출한 속성으로 제일 많이 발생
 - 설계[Designed] : 규칙화등이 필요해 만든 속성ex_코드성,일련번호
 - 파생[Derived] : 다른 속성들로부터 계산/변형되어 만들어진 속성. 정보가 추가될 때 마다 변경이 잦음.
 ```sql
 CREATE TABLE Student (
    StudentID INT PRIMARY KEY, -- 기본키 속성
    Name VARCHAR(100), -- 일반 속성
    DateOfBirth DATE, -- 일반 속성
    Address VARCHAR(255), -- 일반 속성
    CreatedAt TIMESTAMP, -- 설계 속성
    UpdatedAt TIMESTAMP -- 설계 속성
 );
 ```

** 구성 방식에 따른 분류 **
 - PK [PrimareyKey] : 고유하게 식별할 수 있는 속성. 비어있을 수 없음
 - FK [ForeignKey] : 다른 엔터티의 기본키를 참조하는 속성. 외래키를 통해 엔터티간의 관계를 정의
 - 일반[Regular] : 기본이나 외래키가 아닌 일반속성 ex_이름 주소 전번
 - 복합[Composite] : 두 개 이상의 속성이 결합되어 하나의 속성을 이루는 경우 ex_주소는 도시 도로명 우편번호 등으로 나뉘어짐
 ```sql
 -- 학생(Student) 테이블
 CREATE TABLE Student (
    StudentID INT PRIMARY KEY, -- 기본키
    Name VARCHAR(100),
    DateOfBirth DATE
 );

 -- 수강 기록(Enrollment) 테이블
 CREATE TABLE Enrollment (
    EnrollmentID INT PRIMARY KEY, -- 기본키
    StudentID INT, -- 외래키
    CourseID INT,
    EnrollmentDate DATE,
    FOREIGN KEY (StudentID) REFERENCES Student(StudentID) -- 외래키 제약 조건
 );
 ```
** 도메인 (Domain) **
각 속성이 입력받을 수 있는 값의 정의 및 범위를 의미 (ex_나이 범위를 0~120세정도로 설정하는 것)

### 관계(Relationship)
엔터티 내의 인스턴스들 간에 서로 논리적인 연관성이 있는 상태
 * 존재 : 직원 -소속- 부서
 * 행위 : 고객 -주문- 상품
 * <기출> ERD[Entity-Relationship-Diagram]는 존재/행위 등 관계를 위처럼 똑같이 표시
 ```SQL
 학생(Student)
 -------------------
 StudentID (PK)
 Name
 DateOfBirth
  
 수강(Enrollment)
 -------------------
 EnrollmentID (PK)
 StudentID (FK)
 CourseID (FK)
 EnrollmentDate
 
 강의(Course)
 -------------------
 CourseID (PK)
 CourseName
 ```

 * <암기해라> UML 클래스다이어그램 : 연관관계(존재)는 실선, 의존관계(행위)는 점선
** 관계를 표시하는 방법 ** <중요함>
   * IE : 
   * BARKER
 1. 관계명[Membership] : 엔터티간의 관계 유형. 주로 부모-자식 관계로 구분, 상속관계를 표시
 2. 관계차수[Cardinality/Degree] : 각 인스턴스들이 얼마나 참여하는지 의미
 3. 관계선택사양[Optionality] : 한 엔터티의 존재 여부가 필수적인지? ex_주문은 반드시 고객을 필요/직원은 선택적으로 부서에 소속

* <키워드 암기>관계 체크 사항 - 현재 설정한 관계가 적합한지 보는 체크리스트
 1. 두 엔터티 사이에 관심 있는 연관규칙이 있는가?
 2. 두 엔터티 사이에 정보의 조합이 발생하는가?
 3. 업무기술서, 장표에 관계연결에 대한 규칙이 서술 되었는가?
 4. 업무기술서, 장표에 관꼐연결을 가능케 하는 동사가 있는가?<명사로 기출>

## 데이터 모델링
 추상화 단순화 명확화 하여 일정표기법으로 표시하는 것.
 * 개발순서 : 기획(요구사항) -> 설계 -> 개발 -> 테스트 -> 운영
 1. 개념적 데이터 모델링 : 추상적 업무중심적 포괄적 전사적(EA) DBMS독립적 ERD생성
 2. 논리적 데이터 모델링 : KEY 속성 관계등을 정의 <중요>재사용성 높음 특정 DBMS에 종속적 정규화수행
 3. 물리적 데이터 모델링 : 실제DB에 이식되도록 성능, 저장 등 물리적 성격 고려 구체적 수준

 
 * ERD[EndtityRelationshipDiagram] : 엔터티와 엔터티간의 관계를 발견하고 이를 그림으로 표현하는 행위. 개념적 데이터 모델링의 결과물
 ### 개념적 데이터 모델링
 * 순서<기출> 
   1. 엔터티를 그린다. 보통 중요한 엔터티를 왼쪽 상단에 둔다
   2. 엔터티를 적절히 배치한다.
   3. 관계를 설정한다.
   4. 관계명을 기술한다. (생략되는 경우가 많음)
   5. <중요>관계의 참여도(Cardinality)를 기술
   6. 필수여부[Optionality]기술 

 ### 논리적 데이터 모델링
 <중요>재사용성이 높음
 * 순서
   1. 속성을 아래로 모두 입력한다.
   2. 각 엔터티에서 식별자[PrimaryKey]를 선택
   3. 정규화를 수행해 중복되는 데이터 입력이 없도록 한다. (중복이 없더록 엔터티를 쪼갬)
   4. 관계설정[foreignKey]을 한다. : 정규화로 쪼개진 대상들을 연결하기 위해 식별자를 빌려온다.

 ### 데이터 모델링의 중요성
 1. 파급효과 : 초반이 엉망이면 나중에 힘들어짐
 2. 간결한 표현 : 서로 의사소통을 위해 모델링된 설계도로도 쉽게 이해가 가능
 3. 데이터 품질 : 이상한 데이터 방지(중복방지-정규화, 비유연성 - 2개의 연락처 추가??, 비일관성 - 변경된 연락처 동기화)

 ### 데이터베이스 3단계 구조
 <종류와 특징 알기/기출>
 1. 외부[External]스키마 : 여러 사용자 각각의 관점
 2. 개념[Conceptual]스키마 : 통합적, 조직 전체의 DB관점
 3. 내부[Internal]스키마 : 데이터 물리 저장구조 표현

 ### 식별자
 엔터티 내 유일하게 인스턴스를 식별할 수 있는 속성의 집합
 * 분류
   1. 대표성 여부 : 주 식별자 VS 보조식별자
   2. 스스로 생성 여부 : 내부식별자 VS 외부식별자
   3. 단일속성여부 : 단일식별자 VS 복합식별자
   4. 대체여부 : 본질식별자(ex_토지개발부서) VS 인조식별자(ex_CODE001)
 * <중요>주식별자 도출기준과 특징
   1. 유일성 : 인스턴스를 유일하게 식별
   2. 최소성 : 속성의 수가 많지 않기
   3. 불변성 : 가급적 변하지 않기
   4. 존재성 : 값이 꼭 입력되기

 ## 1-2 데이터 모델과 SQL

 ### 정규화
 데이터에 대한 중복을 제거하고 데이터가 관심가별로 처리되도록 엔터티를 쪼개 성능을 향상 시키는 활동
 *정규화 과정 : 비정규형 --(도메인원자성)--> 1NF(1차정규화) --(부분종속성제거)--> 2NF --(이행종속성제거)--> 3NF --(결정자제거)--> BCNF --(다치종속)-->  4NF --(조인속성)--> 5NF
 3NF까지만 알고있어라

 * 1차 정규화
 엔토티에서 다중 속성이 있거나 하나의 속성에 여러 속성값이 있으면 원자값으로 쪼개준다.
 ex_주소1,주소2/연락처 2개면 한 줄 에 있으면 나누어라.

 * 2차 정규화
 1차 정규화를 만족하면서, 부분 함수 종속이 없어야 함. 기본키의 일부분에만 종속된 속성이 없어야 함.

 * 3차 정규화
 엔터티에서 일반속성끼리 함수종속이 발생하면 쪼개줍니다.ex_삼단논법생각 X가Y식별 Y가 z식별시 X가Z식별함.이게 이행종속이다.

 * 반정규화(역정규화)
 정규화한 데이터를 다시 합쳐서 중복되게 하거나 통합, 분리 등을 수행하는 모델링 방식
